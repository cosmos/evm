# Cosmos EVM v0.6.x → v0.7.0 Migration

v0.7.0 ships **parallel EVM execution** via the Krakatoa mempool and BlockSTM groundwork. Most of the migration work is plumbing the new keeper signatures and store keys these features require. Secondary breaking changes — IBC callback hardening, EIP-712 verification, the `x/precisebank` deprecation — round out the upgrade.

For deeper background on the new features, see:

- Krakatoa overview — _TODO: link_
- BlockSTM and parallel execution — _TODO: link_
- Virtual fee collection — _TODO: link_

If you're skipping a release, read the [v0.5.x → v0.6.0 guide](./v0.5.x_to_v0.6.0.md) first — its StateDB / `callFromPrecompile` API break still applies.

This guide is split into two parts: **Part 1** is the action checklist for a chain operator / app developer migrating their fork. **Part 2** is the background — what's changing under the hood, why it's a state break, and what to test for before shipping.

# Part 1 — What you need to do

## Step 1: Bump dependencies

```bash
# in your go.mod
github.com/cosmos/evm v0.7.0
```

```bash
go mod tidy
```

v0.7.0 transitively bumps `github.com/ethereum/go-ethereum` to v1.17 (#1122). If you depend on geth APIs directly (custom tracers, custom precompiles importing `core/vm`), expect compile errors and follow the geth 1.16 → 1.17 changelog for shim updates.

## Step 2: Update store keys in `app.go`

Replace the transient store keys with object store keys, and build a `nonTransientKeys` slice for the EVM keeper:

```diff
  keys := storetypes.NewKVStoreKeys(/* ... */)
- tkeys := storetypes.NewTransientStoreKeys(evmtypes.TransientKey, feemarkettypes.TransientKey)
+ oKeys := storetypes.NewObjectStoreKeys(banktypes.ObjectStoreKey, evmtypes.ObjectKey)
+
+ var nonTransientKeys []storetypes.StoreKey
+ for _, k := range keys {
+     nonTransientKeys = append(nonTransientKeys, k)
+ }
+ for _, k := range oKeys {
+     nonTransientKeys = append(nonTransientKeys, k)
+ }
```

Update the `EVMD` struct field:

```diff
- tkeys map[string]*storetypes.TransientStoreKey
+ oKeys map[string]*storetypes.ObjectStoreKey
```

Mount object stores instead of transient stores:

```diff
- app.MountTransientStores(tkeys)
+ app.MountObjectStores(oKeys)
```

Wire the bank keeper's object-store key:

```diff
+ app.BankKeeper = app.BankKeeper.WithObjStoreKey(oKeys[banktypes.ObjectStoreKey])
```

## Step 3: Update keeper constructors in `app.go`

### `feemarketkeeper.NewKeeper` — drop transient key

```diff
  app.FeeMarketKeeper = feemarketkeeper.NewKeeper(
      appCodec, authtypes.NewModuleAddress(govtypes.ModuleName),
      keys[feemarkettypes.StoreKey],
-     tkeys[feemarkettypes.TransientKey],
  )
```

### `transferkeeper.NewKeeper` — inline address codec, drop ICS4Wrapper / duplicate ChannelKeeper

```diff
  app.TransferKeeper = transferkeeper.NewKeeper(
      appCodec,
+     evmaddress.NewEvmCodec(sdk.GetConfig().GetBech32AccountAddrPrefix()),
      runtime.NewKVStoreService(keys[ibctransfertypes.StoreKey]),
-     nil, // ICS4Wrapper param
-     app.IBCKeeper.ChannelKeeper,
      app.IBCKeeper.ChannelKeeper,
      app.MsgServiceRouter(),
      app.AccountKeeper,
      app.BankKeeper,
      authAddr,
  )
- app.TransferKeeper.SetAddressCodec(evmaddress.NewEvmCodec(sdk.GetConfig().GetBech32AccountAddrPrefix()))
```

### `evmkeeper.NewKeeper` — object store keys, BankKeeper replaces PreciseBankKeeper

```diff
  app.EVMKeeper = evmkeeper.NewKeeper(
-     appCodec, keys[evmtypes.StoreKey], tkeys[evmtypes.TransientKey], keys,
+     appCodec, keys[evmtypes.StoreKey], oKeys[evmtypes.ObjectKey], nonTransientKeys,
      authtypes.NewModuleAddress(govtypes.ModuleName),
      app.AccountKeeper,
-     app.PreciseBankKeeper,
+     app.BankKeeper,
      app.StakingKeeper,
      app.FeeMarketKeeper,
      &app.ConsensusParamsKeeper,
      &app.Erc20Keeper,
      evmChainID,
      tracer,
  )
```

### `DefaultStaticPrecompiles` — Bank, value-typed TransferKeeper, ClientKeeper

```diff
  precompiletypes.DefaultStaticPrecompiles(
      *app.StakingKeeper,
      app.DistrKeeper,
-     app.PreciseBankKeeper,
+     app.BankKeeper,
      &app.Erc20Keeper,
-     &app.TransferKeeper,
+     app.TransferKeeper,
      app.IBCKeeper.ChannelKeeper,
+     app.IBCKeeper.ClientKeeper,
      app.GovKeeper,
      app.SlashingKeeper,
      appCodec,
  )
```

`IBCKeeper.ClientKeeper` is required because it backs the new ICS-02 client-router precompile (#768).

### `erc20keeper.NewKeeper` — Bank, value-typed TransferKeeper

```diff
  app.Erc20Keeper = erc20keeper.NewKeeper(
      keys[erc20types.StoreKey],
      appCodec,
      authtypes.NewModuleAddress(govtypes.ModuleName),
      app.AccountKeeper,
-     app.PreciseBankKeeper,
+     app.BankKeeper,
      app.EVMKeeper,
      app.StakingKeeper,
-     &app.TransferKeeper,
+     app.TransferKeeper,
  )
```

## Step 4: Decide on opt-ins (consensus-affecting)

Two opt-ins control whether your chain gets the throughput benefits of v0.7. **All validators must agree** on these settings — they affect the state-transition function. Pick once for your release.

### Optimistic execution (recommended)

Append to your baseapp options at app construction:

```diff
+ baseAppOptions = append(
+     baseAppOptions,
+     baseapp.SetOptimisticExecution(),
+ )

  bApp := baseapp.NewBaseApp(/* ... */, baseAppOptions...)
```

### Virtual fee collection (recommended for 18-decimal chains)

After `WithStaticPrecompiles`, opt the EVM keeper into the parallel-safe virtual fee collection path:

```diff
  app.EVMKeeper = evmkeeper.NewKeeper(/* ... */).WithStaticPrecompiles(/* ... */)
+ app.EVMKeeper.EnableVirtualFeeCollection()
```

> ⚠️ **Do not call `EnableVirtualFeeCollection()` if your gas token is not 18-decimal.** The virtual collector assumes 18-decimal precision; calling it on, e.g. a 6-decimal denom will panic at fee deduction. See Step 5.

## Step 5: Drop `x/precisebank` (18-decimal chains)

`x/precisebank` is deprecated (#1019). For 18-decimal chains, remove it entirely:

```diff
  type EVMD struct {
      // ... other keepers
-     PreciseBankKeeper precisebankkeeper.Keeper
  }

- // Set up PreciseBank keeper
- app.PreciseBankKeeper = precisebankkeeper.NewKeeper(/* ... */)

  app.ModuleManager = module.NewManager(
      // ... other modules
-     precisebank.NewAppModule(app.PreciseBankKeeper, app.BankKeeper, app.AccountKeeper),
  )

- func (app *EVMD) GetPreciseBankKeeper() *precisebankkeeper.Keeper { /* ... */ }
```

If you previously injected `app.PreciseBankKeeper` into custom modules, swap to `app.BankKeeper` and verify decimal expectations at each call site.

**For non-18-decimal chains:** pin `cosmos/evm/contrib/x/precisebank` and **do not** call `EnableVirtualFeeCollection()`. See [Part 2 — Why precisebank is deprecated](#why-precisebank-is-deprecated) for the rationale and your longer-term options.

## Step 6: Update custom code

### Custom ante handlers / mempool plugins

Anything reading `TransientGasUsed` no longer compiles. Read gas-wanted from the SDK context:

```diff
- ctx, gasUsed := getTransientGas(ctx)
+ gasUsed := ctx.GasMeter().GasConsumed()
```

If your decorator wrote to the feemarket transient store, remove that logic — feemarket transient state is gone.

If you've extended the mempool with a custom subpool, expose a `RecheckTxFn` to participate in Krakatoa's promote/demote rechecks:

```go
func (p *MyPool) RecheckTxFn() mempool.RecheckFn {
    return p.recheckFn
}
```

Returning nil opts your subpool out of rechecks, leaving stale txs in the pool when state advances.

### Custom indexers / receipt consumers

- Read `log.Index` and `log.TxIndex` only after post-execution patching (#1132). Recomputing from raw event order will diverge from canonical receipts.
- Drop any workaround that special-cased the `MaxUint64` overflow on `transactionIndex` — fixed in #1047.
- Don't assume tx counts in receipt arrays match raw block tx counts. StateDB-error txs are now skipped during receipt conversion (#1107).

### Mocks of `EVMKeeper`

Regenerate against v0.7.0 — the constructor signature (Step 3) and static-precompile dependencies (Step 3) shifted.

## Step 7: Coordinate the upgrade

This is a normal Cosmos consensus-breaking upgrade.

- **All validators must run the same v0.7 binary post-upgrade.** Mixing v0.6 and v0.7 binaries will fork.
- **All validators must agree on the Step 4 opt-ins.** Virtual fee collection in particular changes intermediate state during a block; validators that disagree will fork.
- **There is no on-chain governance flag** for parallel execution. Toggling opt-ins later requires a new binary release and another `MsgSoftwareUpgrade`.

The reference upgrade handler is registered as `v0.6.0-to-v0.7.0` in `evmd/upgrades.go`. It runs `RunMigrations` with empty `StoreUpgrades` — no module schema changes. Submit a `MsgSoftwareUpgrade` with `name: "v0.6.0-to-v0.7.0"` at your target height and swap the binary at the halt height.

## Step 8: Verify

```bash
go build ./...
go test ./...
```

### Required pre-release verification

These are not optional. BlockSTM correctness depends on every conflicting state read/write being declared to the scheduler, and any custom module that bypasses the standard keeper interfaces can introduce nondeterminism. See [Part 2 — Pre-upgrade verification](#pre-upgrade-verification) for what to look for and why.

- [ ] **Multi-node apphash test.** ≥3 validators, contended-load workload (see Part 2), every node agrees on every block's apphash.
- [ ] **Replay determinism.** Run the same block stream against v0.7 ≥6 times — state roots must be bit-identical.
- [ ] **v0.6 → v0.7 diff.** Replay the same block stream on both binaries and confirm the post-upgrade state-root diff is fully explained by the documented semantic changes.
- [ ] **Contended-account stress test.** Workload that has many txs touching the same account / contract per block.
- [ ] **Custom-module audit.** Look for raw store access, package-level globals, or transient state in your forks of ante handlers, message handlers, and EndBlockers.
- [ ] **EIP-712 wallet integration.** Re-validate end-to-end if you have wallets signing EIP-712 (#860).
- [ ] **IBC src-callback negative cases.** #993 enforces contract == sender; #1061 blocks nested ICS20 forwarding.

### Sanity check the upgrade path

```bash
make test-system
```

Builds a v0.6.0 legacy binary, drives a `MsgSoftwareUpgrade` for `v0.6.0-to-v0.7.0`, swaps to the current binary at the halt height, and runs a smoke `bank send`. This validates the upgrade path but does **not** exercise contended-account scenarios — it's necessary, not sufficient.

# Part 2 — What's changing and why

## Krakatoa & BlockSTM

v0.7.0 enables parallel EVM transaction execution by removing the transient-store-based serial-execution blockers and replacing them with object-store keys safe under speculative re-execution (#589, #1030).

For full feature documentation see _TODO: link to Krakatoa docs_.

The pieces that landed in v0.7:

- **Per-tx object stores** (`evmtypes.ObjectKey`, `banktypes.ObjectStoreKey`). Block bloom contributions, log indexing, and gas accounting are keyed per-tx and reduced into block-level results at `EndBlock`.
- **`TransientGasUsed` removed.** Gas-wanted now flows through the SDK context.
- **Post-execution log index patching** (#1132). Block-cumulative `log.Index` and eth-only `log.TxIndex` are assigned after parallel execution settles.
- **Incarnation cache for verify-result** (#1082). Signature/auth verification results are cached per-incarnation across BlockSTM re-execution attempts.
- **Krakatoa mempool** (#1030, #1053, #1112) replaces the legacy EVM mempool path. Supports sudo-rechecking pooled txs through the AnteHandler on promote/demote, reapable-list filtering, and pool-level lifecycle hooks.

The `operate_exclusively` toggle introduced in #1060 was removed in #1112 once Krakatoa was made the only mempool — there is no longer a runtime flag controlling this.

### Always-on vs opt-in

The plumbing is always-on; the throughput-unlocking layers on top are opt-in:

| Change                                         | Always-on after v0.7 | Where you control it                              |
| ---------------------------------------------- | -------------------- | ------------------------------------------------- |
| Object-store migration for gas / bloom / index | ✅ always-on         | n/a (PR #589 plumbing)                            |
| Krakatoa mempool replaces legacy EVM mempool   | ✅ always-on         | n/a (PR #1112 made it exclusive)                  |
| Virtual fee collection                         | ❌ opt-in            | `app.EVMKeeper.EnableVirtualFeeCollection()`      |
| Baseapp optimistic execution                   | ❌ opt-in            | `baseapp.SetOptimisticExecution()` in app options |

If you want a more conservative initial rollout, you can ship v0.7 with the always-on changes and skip both opt-ins, then enable them in a later upgrade once you've burned in the always-on path.

## Why this is a state break

Like every Cosmos EVM minor-version upgrade, v0.7 is consensus-breaking and requires a coordinated `MsgSoftwareUpgrade`. Concretely:

- The semantic API changes (see [Other breaking semantic changes](#other-breaking-semantic-changes)) all change the state-transition function. The same input transactions produce different state roots before and after the upgrade height.
- The transient → object store migration does not migrate any persisted state — transient state is per-block by definition, so it's empty at block boundaries. But validators running different binaries (some on transient, some on object store) at the same height will fork.
- Virtual fee collection, when enabled, settles fees at `EndBlock` rather than per-tx. End-of-block account balances should match the non-virtual path on a well-formed chain, but the in-flight intermediate state differs.

The reference upgrade handler runs `RunMigrations` with empty `StoreUpgrades`, so there are no module schema bumps to coordinate beyond the binary swap itself.

## Pre-upgrade verification

BlockSTM correctness is a contract: every conflicting state read/write must be declared to the scheduler so it can detect conflicts and re-execute as needed. A custom module, ante decorator, or precompile that performs a "hidden" state access — reading or writing through a non-tracked store handle, a package-level global, or directly to `tkeys` — can produce nondeterministic state under parallel execution.

Symptoms of a determinism bug typically look like:

- Lost balance writes when contended accounts are read concurrently.
- Apphash divergence across validator nodes processing the same block stream.
- Replays producing different state roots run-over-run on the same input.

### What "contended-account scenarios" means

A **contended account** is an account or contract whose state is touched by multiple transactions in the same block. Examples:

- An ERC-20 token contract where many txs in one block transfer between different pairs — every tx reads and writes the contract's balance map.
- A DEX pool, AMM, or lending market where many swap / deposit / withdraw txs in one block all read and write the pool's reserves.
- A faucet, airdrop, or rewards contract being drained by many recipient txs in one block.
- A single EOA being spent from by multiple txs in one block (legitimate replacement-tx flows, or batched fee payers).
- A global counter, registry, or oracle contract that every tx in a block writes to.

These are the cases where BlockSTM's scheduler must (a) detect that two concurrently-executing txs both touched the same state, and (b) re-execute the later one with the earlier one's effects visible. If the scheduler misses the conflict, you get a lost write — the second tx's view of state was stale, but its result is committed anyway.

A **contended-account stress test** is a synthetic workload that maximizes these conflicts to flush out scheduler bugs and undeclared state accesses. Build one by submitting many txs that all hit the same hot account or contract in a single block, ideally with mixed reads and writes. Run it under multi-node consensus and confirm every validator agrees on the apphash for every block.

### Custom-module audit

For every module not in `cosmos/evm`, audit ante decorators, message handlers, and EndBlockers for state reads or writes that bypass the standard keeper interfaces:

- Raw `KVStore` access using a stored `*storetypes.KVStoreKey` directly (not via a keeper method).
- Package-level globals or singletons used to share state between handlers.
- Transient store usage that wasn't migrated to the object store.
- Cached values held across tx boundaries inside a keeper (e.g. memoization that survives between txs in the same block).

These are the most common sources of BlockSTM nondeterminism. The fix is typically to route the access through a tracked store handle so the scheduler can observe it.

## Why precisebank is deprecated

The `x/precisebank` deprecation (#1019) is tightly coupled to the Krakatoa work: the new virtual fee collection path assumes an 18-decimal gas token. Precisebank's reason for being was to bridge non-18-decimal denoms into the EVM's 18-decimal world, and consolidating around 18-decimal precision is what lets the parallel fee path stay correct without per-tx decimal conversions.

`EvmCoinInfo.Decimals` is also marked deprecated (#1029).

> Going forward, non-18-decimal chains are not fully supported. v0.7.0 still ships precisebank under `contrib/`, but future EVM releases will not test or maintain it.

If you operate a non-18-decimal chain, your options are:

1. Stay on v0.6.x until you can migrate your base denom to 18 decimals.
2. Pin `cosmos/evm/contrib/x/precisebank`, **do not** call `EnableVirtualFeeCollection()`, and accept that future EVM releases may not be tested against this configuration.

## Authority params (#1130)

`x/vm`, `x/erc20`, and `x/feemarket` `MsgServer` handlers now use `sdk.ValidateAuthority`, which consults the cosmos-sdk v0.54 consensus `AuthorityParams` when present and falls back to the per-module module-account authority otherwise. No code change is required if you keep the existing pattern; populate `AuthorityParams` in the consensus module if you want to centralize gov authority.

## JSON-RPC config additions

Additive, defaults are safe — surface in your `app.toml` template if you want to tune:

- `json-rpc.http-body-limit` (#1108, #1109) — request body size cap.
- `json-rpc.filter-cleanup-interval` and related (#1008) — replaces the old hard global filter cap with idle-timeout reclamation. The previous global filter cap is no longer enforced.

## Other breaking semantic changes

These don't move keeper signatures (or change them only marginally) but alter behavior in ways that affect chains and indexers.

### `CallEVMWithData` honors `gasCap` (#992)

Previously ignored the `gasCap` argument and used the default cap. Now respects the value passed. If you were relying on the bug, gas behavior will differ.

### IBC src callbacks: contract == sender (#993)

For IBC acknowledgement and timeout callbacks, the `src_callback` contract address is now enforced to match the packet sender. Arbitrary cross-account contract execution via callbacks is rejected.

### Nested ICS20 forwarding blocked in src callbacks (#1061)

ICS20 packet forwarding originated from within a source callback is blocked. Chain logic that triggers a forward as part of an outgoing-packet callback must be redesigned.

### Access Control List case-insensitive (#1079)

ACL comparisons are now case-insensitive. On-chain ACL entries written with mixed casing that you relied on for de-duplication need to be re-keyed.

### EIP-712 signature verification (#860)

EIP-712 signature verification now uses the configured EVM chain ID (instead of parsing the cosmos chain-id string) and the `aminojson` sign-mode handler (replacing `legacytx.StdSignBytes`). Wallets and relayers signing EIP-712 transactions should be re-validated end-to-end.

### EIP-6780 selfdestruct fix (#905)

Contracts at pre-funded addresses now have their code and account persisted before `DeleteAccount`'s `IsContract` check, so EIP-6780 selfdestruct correctly deletes the contract. Contracts that depended on the prior buggy behavior will see different state after selfdestruct.

### `block.coinbase` opcode (#817, #920)

`GetCoinbaseAddress` now (a) handles the empty-proposer case in contexts like `CheckTx`, and (b) converts the validator operator's Bech32 address into an Ethereum address. The opcode now returns the proposer's derived Ethereum address rather than a zero / mis-encoded value.
