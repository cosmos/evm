# Cosmos EVM v0.6.x ŌåÆ v0.7.0 Migration

v0.7.0 ships two independent headline features ŌĆö the **Krakatoa application-layer mempool** and **BlockSTM parallel execution** ŌĆö plus a large dependency-stack bump (cosmos-sdk v0.54, cometbft v0.39, ibc-go v11, geth 1.17, Go 1.25). Most of the migration work is plumbing the new keeper signatures and store keys these features require.

Headline features:

- **Krakatoa mempool** (#1030, #1053, #1112) ŌĆö moves the EVM mempool out of CometBFT and into the application layer. The chain now has full control over `CheckTx` / `RecheckTx`, sudo-rechecking on promote/demote, reapable-list filtering, and pool-level lifecycle hooks. It's a faster, more controllable replacement for the legacy EVM mempool ŌĆö independent of parallel execution.
- **BlockSTM parallel execution** (#589, #1082, #1132) ŌĆö the state-transition function runs txs in parallel under a software-transactional-memory scheduler, with per-tx object stores for bloom / log indexing / gas accounting and an incarnation cache for signature/auth verification. Independent of Krakatoa.
- **Virtual fee collection** (opt-in, 18-decimal chains) ŌĆö fees settle at `EndBlock` rather than per-tx, removing a serial-execution bottleneck under BlockSTM.
- **ICS-02 client-router precompile** (#768) ŌĆö Solidity contracts can now manage IBC light clients.
- **Authority params** (#1130) ŌĆö `x/vm`, `x/erc20`, `x/feemarket` `MsgServer` handlers consult `AuthorityParams` from the consensus module before falling back to per-module gov authority.
- **OpenTelemetry tracing** (#871, #863) ŌĆö spans across `x/vm` and the JSON-RPC layer.
- **JSON-RPC filter lifecycle overhaul** (#1008) ŌĆö idle-timeout reclamation replacing the old global filter cap.

Secondary breaking changes ŌĆö IBC callback hardening, EIP-712 verification, the `x/precisebank` deprecation, the `abi.json` strict-ABI requirement (#758) ŌĆö round out the upgrade.

## Step 2: Update store keys in `app.go`

Replace the transient store keys with object store keys, and build a `nonTransientKeys` slice for the EVM keeper:

```diff
  keys := storetypes.NewKVStoreKeys(/* ... */)
- tkeys := storetypes.NewTransientStoreKeys(evmtypes.TransientKey, feemarkettypes.TransientKey)
+ oKeys := storetypes.NewObjectStoreKeys(banktypes.ObjectStoreKey, evmtypes.ObjectKey)
+
+ var nonTransientKeys []storetypes.StoreKey
+ for _, k := range keys {
+     nonTransientKeys = append(nonTransientKeys, k)
+ }
+ for _, k := range oKeys {
+     nonTransientKeys = append(nonTransientKeys, k)
+ }
```

Mount object stores instead of transient stores:

```diff
- app.MountTransientStores(tkeys)
+ app.MountObjectStores(oKeys)
```

Wire the bank keeper's object-store key:

```diff
+ app.BankKeeper = app.BankKeeper.WithObjStoreKey(oKeys[banktypes.ObjectStoreKey])
```

(The matching `EVMD` struct field rename is in Step 3.)

### IBC callbacks middleware ŌĆö wrap with setter calls

### Wire BlockSTM and hydrate globals

These two lines are what attach the BlockSTM scheduler to the baseapp and let the EVM globals survive a node restart. Without them parallel execution is not enabled and `evmCoinInfo` globals stay nil after a restart, panicking any RPC call before `PreBlock`.

After `LoadLatestVersion` (inside `if loadLatest`), hydrate the EVM globals from KV:

```diff
+ ctx := app.NewContextLegacy(true, cmtproto.Header{
+     Height:  app.LastBlockHeight(),
+     ChainID: app.ChainID(),
+ })
+ vmModule.HydrateGlobals(ctx)
```

(`vmModule` is the `vm.NewAppModule(...)` value bound to a local variable so you can reuse it here.)

Then attach the BlockSTM tx runner to the baseapp before returning:

```diff
+ vmrunner.SetRunner(bApp, txnrunner.NewSTMRunner(
+     txDecoder,
+     nonTransientKeys,
+     min(goruntime.GOMAXPROCS(0), goruntime.NumCPU()),
+     true,
+     func(ms storetypes.MultiStore) string {
+         return app.EVMKeeper.GetParams(sdk.NewContext(ms, cmtproto.Header{}, false, log.NewNopLogger())).EvmDenom
+     },
+ ))
```

Imports to add: `goruntime "runtime"`, `cmtproto "github.com/cometbft/cometbft/proto/tendermint/types"`, `"github.com/cosmos/cosmos-sdk/baseapp/txnrunner"`, `vmrunner "github.com/cosmos/evm/x/vm/runner"`.

### Removed `EVMD` methods

These were deleted; remove any callers:

- `GetTKey`, `GetMemKey` ŌĆö transient/mem stores no longer exist as separate maps.
- `GetAuthzKeeper` ŌĆö the helper was redundant; expose `app.AuthzKeeper` directly if you need it.
- `GetPreciseBankKeeper` ŌĆö module is removed (Step 6).
- `SetClientCtx` and the `clientCtx` field ŌĆö unused.

## Step 5: Decide on opt-ins (consensus-affecting)

Two opt-ins control whether your chain gets the throughput benefits of v0.7. **All validators must agree** on these settings ŌĆö they affect the state-transition function. Pick once for your release. Both are enabled in the reference `evmd/app.go` at HEAD.

### Optimistic execution (recommended)

Already wired in Step 4 via `baseapp.SetOptimisticExecution()`. Remove the line if you prefer to ship v0.7 without it.

### Virtual fee collection (recommended for 18-decimal chains)

After `WithStaticPrecompiles`, opt the EVM keeper into the parallel-safe virtual fee collection path:

```diff
  app.EVMKeeper = evmkeeper.NewKeeper(/* ... */).WithStaticPrecompiles(/* ... */)
+ app.EVMKeeper.EnableVirtualFeeCollection()
```

> ŌÜĀ’ĖÅ **Do not call `EnableVirtualFeeCollection()` if your gas token is not 18-decimal.** The virtual collector assumes 18-decimal precision; calling it on, e.g. a 6-decimal denom will panic at fee deduction. See Step 6.

## Step 6: Drop `x/precisebank` (18-decimal chains)

`x/precisebank` is deprecated (#1019). It existed to bridge non-18-decimal denoms into the EVM's 18-decimal world; the new virtual-fee-collection path consolidates around 18-decimal precision and no longer needs it. `EvmCoinInfo.Decimals` is also marked deprecated (#1029).

> **Non-18-decimal chains are not fully supported going forward.** v0.7.0 still ships precisebank under `contrib/x/precisebank`, but future EVM releases will not test or maintain it. If your gas token is not 18-decimal, either stay on v0.6.x until you can migrate, or pin `cosmos/evm/contrib/x/precisebank` and **do not** call `EnableVirtualFeeCollection()`.

For 18-decimal chains, remove the module entirely:

```diff
  type EVMD struct {
      // ... other keepers
-     PreciseBankKeeper precisebankkeeper.Keeper
  }

- // Set up PreciseBank keeper
- app.PreciseBankKeeper = precisebankkeeper.NewKeeper(/* ... */)

  app.ModuleManager = module.NewManager(
      // ... other modules
-     precisebank.NewAppModule(app.PreciseBankKeeper, app.BankKeeper, app.AccountKeeper),
  )

- func (app *EVMD) GetPreciseBankKeeper() *precisebankkeeper.Keeper { /* ... */ }
```

If you previously injected `app.PreciseBankKeeper` into custom modules, swap to `app.BankKeeper` and verify decimal expectations at each call site.

## Step 7: Update custom code

### Custom ante handlers / mempool plugins

Anything reading `TransientGasUsed` no longer compiles. Read gas-wanted from the SDK context:

```diff
- ctx, gasUsed := getTransientGas(ctx)
+ gasUsed := ctx.GasMeter().GasConsumed()
```

If your decorator wrote to the feemarket transient store, remove that logic ŌĆö `feemarkettypes.TransientKey` and the keeper methods `GetTransientGasWanted`, `SetTransientBlockGasWanted`, `AddTransientGasWanted` are all gone. The same applies to the EVM keeper: `GetBlockBloomTransient`, `SetBlockBloomTransient`, `GetTxIndexTransient`, `SetTxIndexTransient`, and `WithDefaultEvmCoinInfo` were removed.

If you've extended the mempool with a custom subpool, expose a `RecheckTxFn` to participate in Krakatoa's promote/demote rechecks:

```go
func (p *MyPool) RecheckTxFn() mempool.RecheckFn {
    return p.recheckFn
}
```

Returning nil opts your subpool out of rechecks, leaving stale txs in the pool when state advances.

### Custom `BankKeeper` / `BankWrapper` implementations

The interfaces in `x/vm/types/interfaces.go` changed. Any fork that supplies a non-default bank wrapper must update.

`BankKeeper` gained five methods to support virtual fee collection and parallel-safe balance accounting:

```go
SendCoinsFromModuleToAccountVirtual(ctx context.Context, senderModule string, recipientAddr sdk.AccAddress, amt sdk.Coins) error
SendCoinsFromAccountToModuleVirtual(ctx context.Context, senderAddr sdk.AccAddress, recipientModule string, amt sdk.Coins) error
UncheckedSetBalance(ctx context.Context, addr sdk.AccAddress, amt sdk.Coin) error
LockedCoins(ctx context.Context, addr sdk.AccAddress) sdk.Coins
BlockedAddr(addr sdk.AccAddress) bool
```

`BankWrapper` is a hard API break: `MintAmountToAccount` and `BurnAmountFromAccount` were **removed** and replaced by a single `SetBalance(ctx, account, amt *big.Int) error`. Callers and implementations must both change.

`EmitBlockBloomEvent`'s argument type changed from `ethtypes.Bloom` to `[]byte`.

A new `VMKeeper` interface (`GetEvmCoinInfo`) was added; some helpers now take it instead of a concrete keeper.

### Custom indexers / receipt consumers

- Read `log.Index` and `log.TxIndex` only after post-execution patching (#1132). Recomputing from raw event order will diverge from canonical receipts.
- Drop any workaround that special-cased the `MaxUint64` overflow on `transactionIndex` ŌĆö fixed in #1047.
- Don't assume tx counts in receipt arrays match raw block tx counts. StateDB-error txs are now skipped during receipt conversion (#1107).

### Forks of precompile `abi.json`

#758 made the precompile ABI files strict ŌĆö only valid Ethereum ABI fields are accepted. Forks that vendor or override `abi.json` for any precompile must remove non-ABI extension fields, or they will fail to parse at startup.

### Removed helpers

- `Params.GetActiveStaticPrecompilesAddrs()` is gone ŌĆö derive `[]common.Address` yourself if you used it.

### Mocks of `EVMKeeper`

Regenerate against v0.7.0 ŌĆö the constructor signature (Step 4) and static-precompile dependencies (Step 4) shifted.

## Step 8: Coordinate the upgrade

This is a normal Cosmos consensus-breaking upgrade.

- **All validators must run the same v0.7 binary post-upgrade.** Mixing v0.6 and v0.7 binaries will fork.
- **All validators must agree on the Step 5 opt-ins.** Virtual fee collection in particular changes intermediate state during a block; validators that disagree will fork.
- **There is no on-chain governance flag** for parallel execution. Toggling opt-ins later requires a new binary release and another `MsgSoftwareUpgrade`.

The reference upgrade handler is registered as `v0.6.0-to-v0.7.0` in `evmd/upgrades.go`. It runs `RunMigrations` with empty `StoreUpgrades` ŌĆö no module schema changes. Submit a `MsgSoftwareUpgrade` with `name: "v0.6.0-to-v0.7.0"` at your target height and swap the binary at the halt height.

### Sanity check the upgrade path

```bash
make test-system
```

Builds a v0.6.0 legacy binary, drives a `MsgSoftwareUpgrade` for `v0.6.0-to-v0.7.0`, swaps to the current binary at the halt height, and runs a contended-account workload (many txs hitting the same hot contract per block) so the BlockSTM scheduler is exercised across the upgrade boundary.
